# Ground Rules

> 이 문서는 Next Home을 개발할 때 반드시 지켜야 하는 작업 규칙을 정리한다.

## 문서 업데이트는 모든 작업의 완료 조건이다

- 코드 작업이 끝날 때마다 변경된 기능과 구조에 맞춰 관련 문서를 함께 수정한다.
- 문서 업데이트가 끝나지 않은 작업은 완료된 것으로 간주하지 않는다.
- 기능이 추가되거나 변경되면 `features.md`, `roadmap.md`, `todo.md`를 확인한다.
- 시스템 구조나 데이터 흐름이 변경되면 `architecture.md`를 확인한다.
- 테이블이나 필드가 변경되면 `database.md`를 확인한다.
- 폴더나 모듈 구조가 변경되면 `folder-structure.md`를 확인한다.
- 중요한 제품·기술 결정과 그 이유는 `decision-log.md`에 기록한다.
- 구현 여부가 정해지지 않은 제안은 `ideas.md`에 기록한다.
- 서비스의 목적이나 핵심 방향이 변경되면 `vision.md`를 확인한다.

## 작업 완료 전 확인 사항

모든 작업은 다음 순서로 마무리한다.

1. 요구사항을 검증할 실패하는 테스트를 먼저 작성한다.
2. 테스트가 의도한 이유로 실패하는지 확인한다.
3. 테스트를 통과시키는 최소한의 코드를 작성한다.
4. 모든 테스트가 통과하는 상태에서 코드를 리팩터링한다.
5. 코드와 실제 동작에 맞춰 관련 문서를 업데이트한다.
6. 전체 테스트를 다시 실행한다.
7. Git 변경 목록을 확인해 비밀키와 불필요한 파일이 포함되지 않았는지 검사한다.
8. 구현 내용, 테스트 결과, 변경한 문서를 사용자에게 알린다.

## TDD 원칙

- 모든 신규 기능과 버그 수정은 테스트를 먼저 작성하는 TDD 방식으로 진행한다.
- 기본 사이클은 `Red → Green → Refactor`다.
- Red 단계에서는 요구사항을 표현하는 테스트를 작성하고 실패를 확인한다.
- Green 단계에서는 테스트를 통과시키는 데 필요한 최소 구현만 작성한다.
- Refactor 단계에서는 테스트가 통과하는 상태를 유지하며 구조와 이름을 개선한다.
- 테스트하기 어려운 코드는 책임을 나누고 의존성을 분리한다.
- 테스트 없이 작성된 기능은 완료된 것으로 간주하지 않는다.
- 테스트 실행이 불가능한 경우 그 이유와 검증하지 못한 범위를 작업 결과에 명시한다.

### 테스트 범위

- 급지, 평균 평단가, 가격 격차, 추천 등 핵심 계산은 단위 테스트로 검증한다.
- 데이터베이스 쿼리와 PostGIS 공간 연산은 통합 테스트로 검증한다.
- 외부 API 수집기는 실제 API와 분리된 응답 fixture 또는 mock으로 검증한다.
- 브라우저 E2E도 외부 API 요청을 라우팅하거나 대역으로 차단한 상태에서 실행한다.
- 실제 외부 지도 스모크 테스트는 `LIVE_E2E=1`로 명시적으로 실행하고 호출 횟수를 작업 결과에 기록한다.
- 주요 사용자 흐름은 프론트엔드 컴포넌트 테스트와 필요한 최소 E2E 테스트로 검증한다.
- 단순 구현 개수보다 핵심 규칙과 실패 가능성이 큰 경로를 우선 테스트한다.

## 비밀정보 관리

- API 키, DB 비밀번호, 인증정보는 `.env`에서 관리한다.
- `.env`는 Git에 커밋하지 않는다.
- 저장소에는 실제 값이 없는 `.env.example`만 공유한다.
- 외부에 노출된 인증정보는 운영 전에 재발급한다.

## 외부 API 비용 관리

- 개발 단계의 외부 API는 무료 제공량 안에서만 사용한다.
- 네이버 Maps는 대표 계정의 무료 제공량이 적용되는지 확인한다.
- Dynamic Map 한도는 일 10,000건, 월 100,000건으로 설정한다.
- Geocoding 한도는 일 1,000건, 월 20,000건으로 설정한다.
- 애플리케이션은 안전 여유를 두고 Geocoding을 일 900건, 월 18,000건에서 먼저 차단한다.
- 사용량 70%부터 이메일 알림을 받도록 통보 대상을 등록한다.
- Static Map, Reverse Geocoding, Directions 등 사용하지 않는 API는 활성화하지 않는다.
- Geocoding 결과는 PostGIS에 저장하여 같은 주소를 반복 호출하지 않는다.
- 유료 사용이 필요해지면 사용자 승인 없이 한도를 높이지 않는다.
- 네이버 콘솔 한도를 최종 차단선으로 사용하고 코드 내부에도 별도의 더 낮은 호출 예산을 둔다.
- 코드 내부 호출 예산에 도달하면 해당 외부 API 호출을 즉시 중단한다.
- 프론트엔드 브라우저의 카운터나 제한만으로 비용을 통제하지 않는다. 브라우저 제한은 사용자가 우회하거나 초기화할 수 있다.
- Geocoding은 백엔드만 호출할 수 있으며 프론트엔드에서 직접 호출하지 않는다.
- Dynamic Map은 등록된 도메인에서만 로드하고 화면당 지도 인스턴스를 하나만 유지한다.
- 사용량을 늘리기 전에 캐시, 중복 제거, 사전 수집 데이터 활용 등 호출 절감 방법을 먼저 검토한다.
- 네이버 지도 데이터의 저장이나 캐시는 약관상 허용 범위를 확인한 경우에만 적용한다.

## 구현 원칙

- `ideas.md`는 향후 후보 아이디어를 보관하는 문서이며, 사용자가 별도로 구현을 요청하기 전에는 개발 범위에 포함하지 않는다.

- 급지 지도를 서비스의 메인 화면으로 유지한다.
- 급지는 평균 평단가를 기준으로 계산한다.
- 갈아타기 추천은 현재 급지를 기준으로 1급지 위와 2급지 위를 보여준다.
- 개인별 대출, 보유 자금, 필요한 추가 자금은 계산하지 않는다.
- 가격과 격차 데이터를 객관적으로 보여주고 최종 판단은 사용자에게 맡긴다.
- 갈아타기 알림은 브라우저 웹 알림으로 제공한다.
